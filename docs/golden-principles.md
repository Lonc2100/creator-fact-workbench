# 黄金原则与复杂度控制

## 来源

本文件吸收三类原则：

- OpenAI Harness Engineering：短 `AGENTS.md`、仓库知识作为系统记录、机械化约束、熵清理。
- Lennith/MyCodexSkills 的 code-slim-refactor 思路：收敛重构、保持当前契约、验证后再继续。
- 常见工程原则：KISS、YAGNI、DRY、单向依赖、行为保持。

## 黄金原则

### 1. AGENTS 是地图，不是百科全书

`AGENTS.md` 只保留项目根、禁区、核心读序、固定路线和工作规则。细节放入 `docs/`，由任务按需读取。

### 2. 当前契约优先

重构和清理不能为了让测试好过而偷偷改业务含义。先确认当前要保留的契约，再改结构。

### 3. 小步收敛

优先做小的、可验证的改动。每次清理都要说明：

- 移除了什么复杂度；
- 保留了什么行为；
- 跑了什么验证；
- 还有什么风险。

### 4. 删除优于兼容包袱

当前产品不需要的旧路径、旧文档、旧上下文，应进入清理流程。不要为了“也许以后用”保留污染源。

### 5. 边界处验证，内部用类型

外部数据进入系统时必须被 provider 转成内部类型。不要 YOLO 式猜字段，也不要让外部数据格式流进 Service/UI。

### 6. 抽象必须还债

新增抽象要减少真实重复、隔离真实变化，或让测试和审查更容易。否则不加。

### 7. 工作流不替代产品价值

Agent、自动化、观测、看板都只是为了提升自媒体工作台质量。任何治理能力如果不能服务主链路，就延后。

### 8. 审查结论要可执行

Auditor 不能只说“建议优化”。必须指出文件、风险、验收方式和下一步。

## Slim Refactor 模式

### Quick

用于小范围清理：

1. 说明目标、范围、要移除的复杂度。
2. 只改范围内文件。
3. 保持当前契约。
4. 跑相关验证。
5. 报告结果和剩余风险。

### Strict

用于阶段性清理或风险较大的重构：

1. 先写计划。
2. 建立任务清单。
3. 每轮改动后审查。
4. 出现测试/契约不一致时，先做 mismatch 判断。
5. 完成后写报告。

## 复杂度红线

出现以下情况要停下来清理：

- 同一概念出现两个以上来源；
- 一个文件同时承担 UI、业务规则、外部调用；
- Service 直接调用外部工具；
- UI 直接处理外部数据格式；
- 新增工具没有验收命令；
- 文档说法和代码行为不一致；
- Agent 必读文档超过当前任务需要。
## 熵扫描入口

定期清理先运行 `npm run scan:entropy`，再阅读 `.local/entropy-governance-scan/report.md` 和 `docs/entropy-governance.md`。扫描只读，不删除、不迁移、不写真实 DB；所有可删除项只能进入候选清单，等待用户逐项确认。
