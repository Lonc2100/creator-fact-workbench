# 黄金原则与复杂度控制

## 来源

本文件吸收三类原则：

- OpenAI Harness Engineering：短 `AGENTS.md`、仓库知识作为系统记录、机械化约束、熵清理。
- Lennith/MyCodexSkills 的 code-slim-refactor 思路：收敛重构、保持当前契约、验证后再继续。
- 常见工程原则：KISS、YAGNI、DRY、单向依赖、行为保持。

## 黄金原则

### 1. AGENTS 是地图，不是百科全书

`AGENTS.md` 只保留项目根、禁区、核心读序、固定路线和工作规则。细节放入 `docs/`，由任务按需读取。

### 2. 当前契约优先

重构和清理不能为了让测试好过而偷偷改业务含义。先确认当前要保留的契约，再改结构。

### 3. 小步收敛

优先做小的、可验证的改动。每次清理都要说明：

- 移除了什么复杂度；
- 保留了什么行为；
- 跑了什么验证；
- 还有什么风险。

### 4. 删除优于兼容包袱

当前产品不需要的旧路径、旧文档、旧上下文，应进入清理流程。不要为了“也许以后用”保留污染源。

### 5. 边界处验证，内部用类型

外部数据进入系统时必须被 provider 转成内部类型。不要 YOLO 式猜字段，也不要让外部数据格式流进 Service/UI。

### 6. 抽象必须还债

新增抽象要减少真实重复、隔离真实变化，或让测试和审查更容易。否则不加。

### 7. 工作流不替代产品价值

Agent、自动化、观测、看板都只是为了提升自媒体工作台质量。任何治理能力如果不能服务主链路，就延后。

### 8. 审查结论要可执行

Auditor 不能只说“建议优化”。必须指出文件、风险、验收方式和下一步。

### 9. 并行会话优先做长闭环

多会话并行的价值不是让用户频繁切换、反复催促，而是让独立 Worker 自己完成一个清晰边界内的长闭环：

- 读完指定上下文和任务约束；
- 执行允许范围内的实现、审计或验证；
- 按任务顺序跑验收命令；
- 失败时先诊断原因、区分任务失败、环境竞争和已知边界；
- 保存必要证据和 handoff；
- 给出下一步可执行建议。

适合并行的长闭环任务包括 docs/status 审计、bundle 验证、截图回归、只读产品复核、明确范围的 UI polish、单平台采集器研究。它们必须减少人工中途介入，而不是把半成品问题抛回主会话。

重型浏览器、Playwright/E2E、会启动 Next 服务或写隔离 sqlite 的命令仍要串行安排，避免端口、`.next`、`NEXT_DIST_DIR`、浏览器会话和数据库竞争。需要并行时，任务必须声明隔离端口、隔离 DB 和不并行的验收命令。

### 10. Worker 用时服务于质量，不服务于拖时间

子会话任务要按工作量分级，并在 handoff 中记录 Started、Finished、Elapsed 和 Workload class。

- `micro`：单文件清理、只读存在性核验、明确小范围修正。可以短于 15 分钟，但 handoff 要说明为什么这是 micro。
- `normal`：常规实现、审计、bundle 归属、文档协议、轻量验证。若低于 15 分钟，Worker 必须追加补扫，例如复查 diff、搜索相邻规范、补风险矩阵、补验证说明；如果继续做会越界，必须在 handoff 说明。
- `long-cycle`：跨文件归属、重型验证、browser/E2E/Next/sqlite gate、平台闭环验收。若低于 15 分钟，默认不合格，除非 handoff 明确说明继续加深会违反范围或重复已完成的验收。

这不是“拖时间式奖惩”。质量机制是：

- 任务太短时，优先合并相邻只读审计或让 Worker 补 handoff 证据；
- normal/long-cycle 低于 15 分钟时，优先做 extra-depth pass；
- 如果补扫会越界、改动业务代码、触发重型 gate 并行竞争或重复破坏性操作，停止补扫并说明原因；
- 主会话用任务拆分质量、证据完整度和验证结果评价 Worker，不按耗时本身奖惩。

### 11. 自动派发不能替代 Orchestrator 判断

自动派发的目标是减少复制提示词和人工推进频次，不是让项目无人监管。当前项目只接受 Level 2：脚本生成 dispatch queue，主会话或固定 Ops 会话读取后再派发。

必须保持：

- PRD > CURRENT-PLATFORM-STATUS > closure handoff > worker handoff；
- 子会话只能提供证据和完成任务包，不能重定义 PRD；
- 删除文件、保存真实平台数据、登录/扫码/验证码、force push、敏感材料、主线 PRD 改动、同文件并发冲突都必须停下找用户；
- live 3200、browser/E2E、sqlite 写入、Next build、daily gate 串行调度；
- queue 文件是候选，不是无限循环开线程的调度器。

## Slim Refactor 模式

### Quick

用于小范围清理：

1. 说明目标、范围、要移除的复杂度。
2. 只改范围内文件。
3. 保持当前契约。
4. 跑相关验证。
5. 报告结果和剩余风险。

### Strict

用于阶段性清理或风险较大的重构：

1. 先写计划。
2. 建立任务清单。
3. 每轮改动后审查。
4. 出现测试/契约不一致时，先做 mismatch 判断。
5. 完成后写报告。

## 复杂度红线

出现以下情况要停下来清理：

- 同一概念出现两个以上来源；
- 一个文件同时承担 UI、业务规则、外部调用；
- Service 直接调用外部工具；
- UI 直接处理外部数据格式；
- 新增工具没有验收命令；
- 文档说法和代码行为不一致；
- Agent 必读文档超过当前任务需要。

## 熵扫描入口

定期清理先运行 `npm run scan:entropy`，再阅读 `.local/entropy-governance-scan/report.md` 和 `docs/entropy-governance.md`。扫描只读，不删除、不迁移、不写真实 DB；所有可删除项只能进入候选清单，等待用户逐项确认。
