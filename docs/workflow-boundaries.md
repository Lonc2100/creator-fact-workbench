# 每层工作流与边界

## 设计原则

每一层都是一个大项，但每层只做自己的事。这样后面接 Postiz、Mixpost、MediaCrawler、ALwrity、n8n、Metabase、Evidence 时，不会互相打架。

## 1. 信息采集层

负责什么：

- 导入平台后台数据；
- 接入 MediaCrawler 的平台采集结果；
- 接入 n8n 的定时抓取结果；
- 接收人工录入、CSV、JSON。

不负责什么：

- 不直接做最终分析结论；
- 不直接改 UI 状态；
- 不直接发布内容。

参考优秀项目：

- MediaCrawler：学习中文平台数据采集方式。
- n8n：学习自动化调度和任务编排。

工程入口：

- `Providers` 接外部数据。
- `Repo` 保存内部数据。

## 2. 内部存储层

负责什么：

- 把外部信息变成内部统一模型；
- 保存内容、指标、选题、竞品、实验、联系人、复盘、变现线索；
- 保证后续 AI 分析有证据可查。

不负责什么：

- 不保存只存在聊天里的结论；
- 不让外部工具的数据格式污染内部模型。

参考优秀项目：

- Directus / Baserow：学习内容管理后台的数据表组织。
- Metabase：学习数据表如何服务后续分析。

工程入口：

- `Types` 定义实体。
- `Repo` 负责读写。

## 3. 分析与选题层

负责什么：

- 分析内容表现；
- 找爆点、弱点、平台差异；
- 生成选题候选；
- 做竞品拆解和内容策略建议。

不负责什么：

- 不直接发布；
- 不跳过内部数据凭空生成结论。

参考优秀项目：

- ALwrity：学习品牌脑、竞品分析、内容策略结构。
- Evidence：学习把数据结论变成可读报告。

工程入口：

- `Service` 做业务判断。
- AI 分析只能读取内部记录和明确输入。

## 4. 内容生产与发布排期层

负责什么：

- 管理内容队列；
- 管理平台版本；
- 管理草稿、待审核、已排期、已发布状态；
- 后续接自动发布或半自动发布。

不负责什么：

- 不绕过审核直接全自动发；
- 不把平台 API 细节写进业务层。

参考优秀项目：

- Postiz：学习发布日历和内容队列。
- Mixpost：学习多平台发布、平台版本和统计回收。

工程入口：

- `Service` 管状态机。
- `Providers` 接发布工具。
- `UI` 只做审核和操作入口。

## 5. 复盘层

负责什么：

- 生成周复盘；
- 生成月复盘；
- 输出结构化 JSON 和可读 Markdown；
- 给出下阶段行动、表达训练、变现跟进。

不负责什么：

- 不只根据主观聊天生成复盘；
- 不把复盘当成一次性文章，而是要回写为内部记录。

参考优秀项目：

- Evidence：学习 Markdown 数据报告。
- Metabase：学习指标看板和趋势观察。

工程入口：

- `Service` 生成复盘。
- `Repo` 保存复盘结果。
- `UI` 展示和人工确认。

## 6. 质量与审计层

负责什么：

- 检查需求有没有对齐；
- 检查边界有没有越界；
- 检查测试和验收有没有跑；
- 检查是否引入画布上下文污染；
- 检查成本、安全、可靠性。

不负责什么：

- Auditor 不直接改业务代码；
- 不用“感觉完成”代替验收证据。

参考优秀项目：

- Google ADK Evaluation：学习 agent 轨迹和输出评估。
- Azure AI Well-Architected：学习安全、可靠、成本、运营、性能视角。

工程入口：

- `npm run verify:harness`
- `docs/task-board.md`
- `docs/QUALITY_SCORE.md`
- Auditor handoff
