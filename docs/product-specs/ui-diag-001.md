# UI-DIAG-001：当前 UI 病因诊断

## 结论

当前 UI 的问题不是缺组件，而是视觉决策太松：背景、字号、badge、卡片、英文 eyebrow、网格线同时出现，导致页面像“模块堆叠”而不是成熟产品。

## 主要病因

1. **背景噪音过高**
   - 全局网格背景让边框、表格线、卡片线全部叠在一起。
   - 运营后台应该先保证信息读取，背景不应该参与竞争。

2. **字体层级失控**
   - 页面标题、英文 eyebrow、badge、表格标题、卡片标题、说明文字层级过多。
   - 同一页面需要控制在 4-5 个文字层级。

3. **页面结构不够产品化**
   - 发布日历应是“平台行 x 日期列”的排期工具，而不是普通内容卡片流。
   - 右侧详情应是稳定 inspector，不应让主工作区变窄到拥挤。

4. **组件噪音过多**
   - badge、胶囊、阴影、状态色太多，削弱主任务。
   - 状态色只保留给状态，不用于装饰。

5. **参考没有转成验收**
   - Mixpost/Postiz/Metabase/Evidence 只被“口头参考”，没有落成截图级验收标准。

## 重置原则

- 背景：纯净暖白，不再使用全局网格。
- 页面：左侧导航 + 顶部工具条 + 主工作区 + 右侧 inspector。
- 日历：平台作为行，日期作为列，排期卡片落在格子里。
- 文字：中文为主，英文只保留在必要的低权重 label。
- 卡片：轻边框、少阴影、少胶囊。
- 验收：先让 `/calendar` 达标，再迁移内容、看板、复盘。

## 参考落点

- Mixpost/Postiz：发布日历、平台账号、多平台排期。
- Metabase：数据看板的信息层级。
- Evidence：复盘报告、证据块、行动项。
- OpenAI role-based plugins：只借鉴设计角色审查流程，不新增仓库架构。
